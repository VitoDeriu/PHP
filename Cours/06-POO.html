Introduction à la POO

La Programmation Orientée Objet (POO) est un paradigme de programmation qui permet d’organiser le code autour d’entités nommées objets, plutôt qu’autour de fonctions ou de procédures. En PHP, la POO a été introduite dès PHP 4, mais c’est à partir de PHP 5 que le langage a réellement commencé à prendre en charge la POO de manière moderne et complète.

Avantages de la POO :

  • Favorise la réutilisation et la maintenance du code
  • Structure le projet en classes et objets
  • Permet la modularité et la clarté du code
  • Donne la possibilité d’hériter des fonctionnalités déjà existantes

Principes fondamentaux de la POO

Classes et Objets

  • Classe : c’est un plan ou un modèle qui définit les propriétés (variables) et les méthodes (fonctions) associées à un type particulier d’objet.
  • Objet : c’est une instance d’une classe, un exemplaire concret en mémoire.
<?php

class Voiture
{
    public $marque;
    public $couleur;

    public function demarrer()
    {
        echo "La voiture démarre\n";
    }
}

// Instanciation d’un objet
$maVoiture = new Voiture();
$maVoiture->marque = "Renault";
$maVoiture->couleur = "Rouge";
$maVoiture->demarrer();

Propriétés et Méthodes

  • Propriétés : variables déclarées dans le corps d’une classe
  • Méthodes : fonctions déclarées dans le corps d’une classe

Dans l’exemple ci-dessus, $marque et $couleur sont des propriétés, et demarrer() est une méthode.

Visibilité : public, private, protected

  • public : accessible partout (à l’intérieur comme à l’extérieur de la classe)
  • private : accessible uniquement à l’intérieur de la classe elle-même
  • protected : accessible à l’intérieur de la classe et de ses classes dérivées (héritées)

Exemple :

<?php

class Voiture
{
    private $marque;
    protected $couleur;
    public $modele;

    public function __construct($marque, $couleur, $modele)
    {
        $this->marque = $marque;
        $this->couleur = $couleur;
        $this->modele = $modele;
    }

    private function debugMarque()
    {
        echo "Marque : " . $this->marque . "\n";
    }

    public function afficherInfos()
    {
        // debugMarque() est private, on peut l’appeler depuis l’intérieur de la classe
        $this->debugMarque();
        echo "Couleur : " . $this->couleur . "\n";
        echo "Modèle : " . $this->modele . "\n";
    }
}

Constructeurs et Destructeurs

  • Constructeur : une méthode spéciale appelée automatiquement lors de la création d’un objet (nommée __construct())
  • Destructeur : une méthode spéciale appelée lorsque l’objet est détruit ou que le script s’achève (nommée __destruct())
<?php

class Exemple
{
    public function __construct()
    {
        echo "L’objet Exemple est créé.\n";
    }

    public function __destruct()
    {
        echo "L’objet Exemple est détruit.\n";
    }
}

$ex = new Exemple();
// À la fin du script ou quand $ex n’est plus référencé, __destruct() est appelé.

Héritage

L’héritage permet de créer une nouvelle classe (classe enfant) à partir d’une classe existante (classe parent). La classe enfant hérite des propriétés et méthodes de la classe parent et peut les surcharger ou en ajouter de nouvelles.

<?php

class Vehicule
{
    public $type;

    public function __construct($type)
    {
        $this->type = $type;
    }

    public function circuler()
    {
        echo "Le véhicule circule.\n";
    }
}

class Moto extends Vehicule
{
    public function faireUnWheeling()
    {
        echo "La moto fait un wheeling.\n";
    }
}

$moto = new Moto("2 roues");
$moto->circuler();
$moto->faireUnWheeling();

Polymorphisme, Interfaces et Classes abstraites

  • Polymorphisme : un même nom de méthode peut se comporter différemment selon la classe qui l’implémente.
  • Classes abstraites : classes qui ne peuvent pas être instanciées directement et qui peuvent contenir des méthodes abstraites (non implémentées).
  • Interfaces : définissent un ensemble de méthodes sans implémentation, que les classes qui implémentent l’interface doivent définir.
<?php

interface Affichable
{
    public function afficher();
}

abstract class Animal
{
    abstract public function crier();

    public function seDeplacer()
    {
        echo "L’animal se déplace.\n";
    }
}

class Chat extends Animal implements Affichable
{
    public function crier()
    {
        echo "Le chat miaule.\n";
    }

    public function afficher()
    {
        echo "Je suis un chat.\n";
    }
}

$chat = new Chat();
$chat->crier();
$chat->seDeplacer();
$chat->afficher();

Autoload en PHP

L’autoload permet de charger automatiquement les classes dont on a besoin sans avoir à faire manuellement des require ou include partout dans le code.

__autoload() obsolète

Historiquement, PHP offrait la fonction magique __autoload(). Depuis, cette approche est obsolète et a été remplacée par des mécanismes plus flexibles et plus robustes.

spl_autoload_register()

La fonction spl_autoload_register() permet d’enregistrer une ou plusieurs fonctions d’autochargement.

Exemple basique :

<?php

spl_autoload_register(function ($classe) {
    // Supposons que les fichiers de classe sont stockés dans le dossier "classes"
    require_once __DIR__ . '/classes/' . $classe . '.php';
});

// À partir d’ici, instancier une classe dans le dossier "classes" chargera
// automatiquement son fichier correspondant
$voiture = new Voiture();
$voiture->demarrer();

PSR : PHP Standards Recommandations

Les PSR sont un ensemble de recommandations faisant office de "standard" sur la production de code.

Chaque "numéro" de PSR correspond à une problématique : PSR4 concerne l'autoload des classes en fonction des namespaces.

PSR-4 et organisation des namespaces

Pour des projets plus importants, il est souvent recommandé de suivre la norme PSR-4 pour organiser les classes dans des namespaces et un système de dossiers cohérents.

Exemple d’arborescence (PSR-4) :

src/
  Controllers/
    UserController.php
  Models/
    User.php

<?php
// Fichier: src/Models/User.php

namespace App\Models;

class User
{
    // ...
}

Le namespace App\Models doit correspondre à l’emplacement du fichier (répertoire src/Models), conformément à PSR-4.

Composer

Présentation de Composer

Composer est un gestionnaire de dépendances pour PHP. Il permet :

  • D’installer facilement des bibliothèques tierces (packages)
  • De gérer automatiquement l’autoload de vos classes
  • De mettre à jour les packages en quelques commandes

Installation de Composer

  • Télécharger Composer et l’installer globalement ou localement
    • Sous Linux / macOS, on peut l’installer via la ligne de commande :
      curl -sS https://getcomposer.org/installer | php
mv composer.phar /usr/local/bin/composer
    • Sous Windows, on exécute l’installateur téléchargeable sur le site de Composer.

Utiliser Composer pour l’autoloading

  1. Initialiser un projet Composer :

    composer init

    Cela va créer un fichier composer.json.

  2. Déclarer l’autoload dans composer.json (exemple PSR-4) :

    {
    "autoload": {
        "psr-4": {
            "App\\": "src/"
        }
    }
}

  3. Générer l’autoload :

    composer dump-autoload

    Composer va créer un dossier vendor et des fichiers pour gérer l’autoload.

  4. Inclure l’autoload dans votre projet :

    <?php

require_once __DIR__ . '/vendor/autoload.php';

use App\Models\User;

$user = new User();

Gestion des dépendances

Pour installer un package, on utilise simplement :

composer require vendor/nom-du-package

Par exemple :

composer require nesbot/carbon

Cela ajoute le package dans composer.json et télécharge les fichiers nécessaires dans vendor/.

Exemple complet

Supposons que l’on crée un projet avec la structure suivante :

my_project/
  src/
    Models/
      User.php
    Controllers/
      UserController.php
  composer.json
  index.php

Fichier composer.json

{
  "name": "mon-organisation/mon-projet",
  "autoload": {
    "psr-4": {
      "App\\": "src/"
    }
  },
  "require": {}
}

Fichier src/Models/User.php

<?php

namespace App\Models;

class User
{
    private $name;
    private $email;

    public function __construct($name, $email)
    {
        $this->name = $name;
        $this->email = $email;
    }

    public function getName()
    {
        return $this->name;
    }

    public function getEmail()
    {
        return $this->email;
    }
}

Fichier src/Controllers/UserController.php

<?php

namespace App\Controllers;

use App\Models\User;

class UserController
{
    public function createUser($name, $email)
    {
        // Ici, on utilise la classe User
        $user = new User($name, $email);
        // Traitement éventuel...
        return $user;
    }
}

Fichier index.php (point d’entrée)

<?php

require_once __DIR__ . '/vendor/autoload.php';

use App\Controllers\UserController;

$userController = new UserController();
$newUser = $userController->createUser('Alice', 'alice@example.com');

echo "Utilisateur créé : " . $newUser->getName() . " (" . $newUser->getEmail() . ")";

Après avoir exécuté :

composer dump-autoload

Le projet est opérationnel. Lorsque vous appelez l’URL associée à index.php, toutes les classes se chargeront automatiquement via l’autoload généré par Composer.

Pour résumer

  • La Programmation Orientée Objet rend le code plus structuré et maintenable.
  • L’autoload facilite le chargement des classes sans multiplier les require ou include.
  • Composer est l’outil standard pour gérer les dépendances en PHP et inclut un système d’autoload fiable.
